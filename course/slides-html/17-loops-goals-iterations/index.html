From linear prompts to iterative systems

The prompt is not the unit of design — the loop is.
The thesis

The prompt is not the unit of design — the loop is

Earlier lectures treated the LLM as a function: prompt in, completion out. Here is the turn: almost every serious use is a process, not a single call.

the bigger picture We study the design of iterative agentic systems — a sequence of model calls, tool calls, validations, and revisions around a goal the first call never achieves — not any single tool.
What you will leave with
A vocabulary — goals, agents, skills, hooks, stopping rules — and two worked loops: a transparent root-finder and a nine-agent credit-stress loop.
Why the old way breaks

Why one-shot prompting is fragile

A single prompt is an open-loop controller: its output is used downstream with no channel to correct an error. The model never learns whether it was right.

  1. Errors propagate un-attenuated. One forward pass to parse, extract, compute, and narrate ⇒ an early slip corrupts everything after it, unnoticed.
  2. The spec exceeds one pass. Interacting constraints — turnover, sector caps, tracking error, tax lots — are satisfied jointly, and joint satisfaction falls roughly multiplicatively.
  3. The model cannot see the environment. It guesses the schema, the rate, whether a file exists — acting on a hallucinated world.
the bigger picture No analyst writes a DCF top-to-bottom and submits it. They sketch, sanity-check the terminal value, walk back an over-ceiling margin, re-run, circulate. Expert workflow is intrinsically iterative.
The core mechanism

The goal-directed loop: five phases that repeat

Observe → Propose → Act → Evaluate → Revise, repeating until the goal holds or a stopping rule fires.

  • Observe the state (files, tests, data, prior outputs)
  • Propose an action toward the goal, given that state
  • Act: execute it ⇒ a new state
  • Evaluate: run checks; does the success test hold?
  • Revise: fold the feedback back in, return to Observe
State \(s_t\); goal \(G\) with success predicate \(\mathrm{ok}(\cdot)\). Each step: \[ a_t=\pi(s_t),\quad s_{t+1}=\mathrm{Act}(a_t),\quad e_{t+1}=\mathrm{Eval}(s_{t+1}). \] Revise feeds the concrete evidence \(e_{t+1}\) of what went wrong back into the next Propose — the error-correction channel one-shot prompting lacks.
the Evaluate phase carries most of the value A loop is only as good as its feedback. A weak success test converges happily to something useless. Design the feedback like you design the prompt.
A crucial distinction

Looping vs. just repeating — and when not to loop

Genuine iteration carries information forward; mere repetition just re-rolls the dice. The difference decides whether a loop helps at all.

Genuine iteration: \(a_{t+1}=\pi(s_{t+1},e_{t+1},h_t)\) — the next action depends on the evaluated feedback. If \(a_{t+1}\perp e_{t+1}\), it is mere repetition (sampling), not iteration.
anti-pattern — spurious repetition Re-prompt "fix the failing test" without reading whether it now passes. Same number of calls; no information carried forward; partial edits may leave the code worse.

Loop only if: output is mechanically verifiable; reasoning varies with input; the first attempt is likely wrong and costly; no irreversible harm before a human gate.

02

Goals as control objects

The setpoint of the loop — concrete, checkable, decomposable, and written down.
What a good goal is made of

Concrete, checkable, decomposable — one property per phase

The goal is the loop's setpoint: it governs behavior, decides when to stop, and decides what is acceptable.

  • Concrete (for Act) — names the artifact. "Improve the model" → "raise OOS AUC of credit.py, measured by backtest.py, to \(\geq 0.78\)."
  • Checkable (for Evaluate) — supplies the predicate \(\mathrm{ok}(\cdot)\): "every rubric dimension \(\geq 90\) via /score-content."
  • Decomposable (for Revise) — splits into independently checkable subgoals; localizes failures and bounds context.
a complete goal specifies three distinct things Success criterion (stop because we won) · stopping rule (stop anyway: budget, no-progress) · anticipated failure modes (named in advance, each becomes a check). The common defect: specifying only the first.
Make the goal durable

Write the goal as a file — ideally executable

A goal in a chat prompt vanishes when context rolls over. A goal in a file is durable, auditable, version-controlled — and, when executable, usable directly as the Evaluate phase.

# TASK: Harden the FCF valuation script
## Artifact
valuation/dcf.py  (value_company(ticker) -> float)
## Success criteria
- pytest tests/test_dcf.py exits 0
- Coverage of dcf.py >= 90%
- No division-by-zero on the 12 edge-case tickers
## Stopping rule
- Halt after 6 iterations, or 2 with no new passing tests.
## Anticipated failure modes
- Passing tests by catching all exceptions and returning 0
  -> forbidden; every value finite & positive or raise ValueError.
the decisive move Make "done" and "acceptable" the same code — a validate.sh that exits 0 iff the goal is met, wired into Evaluate and enforced before commit.
A diagnostic

Bad goals vs. good goals — each repair re-enables a phase

Bad goalGood goalRepair adds
"Improve the chapter."Every dimension \(\geq 90\) via /score-content.Checkable predicate

03

Agents, skills, and hooks

The who, the how, and the guardrails that don't depend on the agent.
The who

Agents are specialized iterators — five roles

Useful agents are not chatbots-with-tools; each does one transformation on project state and hands off.

  • Worker — produces/transforms the artifact (the only mandatory role)
  • Critic — structured judgment without rewriting; the natural-language gradient
  • Validator — objective deterministic pass/fail; most often omitted, most missed
  • Planner — decomposes a goal into ordered subtasks
  • Refactorer — improves structure, preserves behavior
critic vs. validator, in finance terms The critic is the senior reviewer who says the model looks aggressive; the validator is the limit system that simply rejects the trade. A loop of only workers + critics can drift confidently in a wrong direction.
central discipline An agent's value is the durable change to state (files, green tests, a commit) — not the text it prints. Measure it by its diff.
The how

Skills are reusable procedures — a skill ≈ a function

If agents are the who, a skill is the how: a named, composable sequence run the same way every time (/draft-chapter, /score-content, /iterate-book-quality).

The skill–function correspondence
A name you invoke it by · scoped inputs · a single responsibility · repeatable (same inputs → same procedure) · composable (one skill calls another: /draft-chapter → /score-content → /refine-until-threshold).
  • Freeze conventions: /new-topic creates chapter/lecture/notebook \(N\) together — numbering cannot drift.
  • Reduce entropy: every invocation pushes a long-running project back toward consistency. The skill library is the project's standard operating procedures.
The guardrail

Hooks: guardrails that don't depend on the agent

A hook runs automatically on an event (pre-tool, post-write, session-end), outside the agent's control. Its defining property: it does not depend on the agent's cooperation.

Pre-action = pre-trade limit check (can block).   Post-action = post-trade surveillance (validate / record / repair / escalate).

#!/usr/bin/env bash
# check-citations.sh -- post-write validation hook
set -euo pipefail
BIB="book/bibliography.bib"; MISSING=0
for f in $(git diff --cached --name-only | grep '\.tex$'); do
  for key in $(grep -oE '\\cite\{[^}]+\}' "$f" \
               | sed -E 's/\\cite\{([^}]+)\}/\1/'); do
    grep -q "{$key," "$BIB" || { echo "Unresolved: $key"; MISSING=$((MISSING+1)); }
  done
done
[ "$MISSING" -gt 0 ] && { echo "FAIL"; exit 1; }; echo "OK"; exit 0
the move A hook converts "is this good?" (opinion) into "compiles, citations resolve, scores \(\geq\) threshold" (a reproducible bit). Hang the stopping condition on the bit, not on the model's confidence.
Making feedback trustworthy

A passing check is necessary, not sufficient

The loop optimizes for the check, not the goal. If the check is easy to pass without meeting the goal, an optimizing loop will find the gap.

The loop optimizes for \(V=\text{pass}\), not the true goal \(G\). Unless \[ \{x: V(x)=\text{pass}\}\subseteq\{x: G\text{ met}\}, \] there exist artifacts that pass yet fail — and an optimizing loop will find them (Goodhart's law).

Make \(V\) hard to satisfy without meeting \(G\): outcome-tied not proxy checks; diverse checks (game them all at once); validator logic out of the worker's reach; a hidden release-time holdout. (Model-risk management's playbook.)

Failure hooks — what fires on a red verdict (escalating)
Block (refuse) · Revert (undo to last-good) · Escalate (hand to a human) · Re-enter (feed the failure back as next input). The operational-risk playbook in miniature.

Designing an iteration loop

A seven-step template, its pseudocode, and why state must live on disk.
The template

Seven steps — and three of them must stay separate

  1. Read state — ground truth from disk + version control first
  2. State the next local goal — a checkable target for this turn
  3. Propose a change (reason)
  4. Apply it (act — write to disk; only now has the world changed)
  5. Validate — a deterministic check; the agent does not grade its own work
  6. Interpret feedback — direction and magnitude of the error
  7. Decide: stop / continue / escalate
two non-negotiables Validation (5), interpretation (6), and decision (7) are separate — collapsing them removes the independent check. And the decision always includes escalate: a loop that can only succeed or grind forever will grind forever.
From template to code

Pseudocode — mapped onto Claude Code primitives

Every line corresponds to a primitive you already know: file reads, a checked-in goal, a skill that validates, a commit that records.

state <- read_state()            # file reads
goal  <- read_goal()             # checked-in goal file
best  <- evaluate(state, goal)
while not goal.satisfied(best) and not goal.budget_exhausted():
    local  <- choose_objective(state, best)
    change <- propose(state, local)     # REASON (ReAct)
    state2 <- apply(change, state)      # ACT: write to disk
    report <- validate(state2)          # a SKILL: deterministic check
    if report.improves_on(best):
        commit(state2, summarize(change, report))   # audit trail
        state, best <- state2, report.score         # accept
    else:
        state <- rollback(state)                     # reject (Self-Refine)
    if stalled(history) or report.requires_human:
        escalate(state, report); break               # hook-gated
emit_report(state, best, history)                     # hook-gated
why this is robust The accept/reject discipline makes the loop robust to a noisy proposer: a bad proposal costs one iteration, not the whole run — because validation and version control are deterministic.
Memory and termination

State lives on disk; stop on a budget

design for amnesia The agent has no durable memory between turns — it is a new employee every morning who reads the files, does a day's work, commits it, and is replaced overnight. Write before you forget.

Persists: files, git history (long-term memory), validation reports, a human-readable status scratchpad. Lost: chain-of-thought, unsaved values. Bonus: all-on-disk state makes the loop resumable.

Stopping rules — exactly one must be guaranteed-terminating
Success · convergence (no gain for \(k\) steps) · budget exhaustion (mandatory) · human-review-required. Convergence/escalation make the loop smart; the budget makes it safe.

Economics: marginal value shrinks near the goal while cost stays flat — stop where marginal value crosses marginal cost (latency binds interactively; dollars bind overnight).

Example 1: derivative-free root finding

The whole machinery, stripped bare — a loop that improves on feedback alone.
The stripped-bare loop

A loop that improves using feedback alone

Find an \(x\) where a function is (nearly) zero, using only the sign and size of the residual — no calculus allowed.

Find \(x\) with \(f(x)=x^3-x-2\approx 0\) to tolerance \(|f(x)|<10^{-3}\) (single root near \(x\approx 1.52\)). The agent may evaluate \(f\) anywhere but is forbidden derivatives — no \(f'(x)=3x^2-1\), no Newton update. It reads only the sign and size of the residual.
finance analogue This is mechanically an IRR or implied-volatility solver. For an IRR, seek \(r\) with \(\mathrm{NPV}(r)=\sum_t c_t/(1+r)^t=0\) — no closed form, so evaluate at trial rates and move by the sign (too-high NPV ⇒ rate too low). Swap \(f\) and the loop is unchanged.

Three artifacts, none containing an algorithm: a goal file (\(|f|<0.001\), \(\leq 12\) iters), a skill (python evaluator.py $x | tee -a results.log), and a stop hook that vetoes "success" unless the last \(|f(x)|<10^{-3}\).

Watching it converge

Iteration trace from a deliberately poor start (x = 0)

Iter\(x\)\(f(x)\)Agent's reasoning
1\(0\)\(-2\)Negative; root to the right. Jump up.
2\(2\)\(4\)Positive; root bracketed in \((0,2)\).
3\(1.5\)\(-0.125\)Negative but small; root just above \(1.5\).
4\(1.6\)\(0.496\)Positive; bracket \((1.5,1.6)\).
5\(1.52\)\(-0.008192\)Tiny negative; nudge up.

The sign change between \(0\) and \(2\) is the engine: a continuous function going \(-\to+\) crosses zero between. The magnitudes (\(|f(1.5)|\ll|f(1.6)|\)) place the root near \(1.5\) (false-position reading). Declaring success at iteration 5 would be vetoed by the hook.

why the inefficiency is the point The zig-zag past the root means the loop reached tolerance without an algorithm — so the same loop works when the residual is a failing test, a style violation, or a reviewer's objection.

Example 2: credit-risk stress-testing loop

From a point estimate to a memo a committee would actually accept.
The real-world loop

A defensible credit-committee memo — not a point estimate

Meridian Components (fictional; figures illustrative): a mid-market manufacturer seeking a \$40m revolver. Looks comfortable — growth, rising EBITDA, leverage in appetite — but cyclical end markets, illiquid inventory, concentrated receivables.

the goal is not "estimate the PD" A point estimate cannot be wrong — nothing fails against it. The goal is a memo a committee would accept: that acceptance is checkable, so the loop runs until the checks pass.
Reconciled in every column: \[ \mathrm{EL} = \mathrm{PD}\times\mathrm{LGD}\times\mathrm{EAD}. \] PD = 1-yr default probability; \(\mathrm{LGD}=1-\text{recovery}\); EAD = drawn balance plus drawdown.

The memo must give PD, LGD, EAD, EL (\$ and bps), drivers, and a recommendation — lend / modify / decline. A "low risk" phrase is gated on both base and stressed EL clearing thresholds.

Oversight, failure modes, and patterns

What the human signature adds, what breaks, and five designs to reuse.
Accountability, not just analysis

Human oversight, sign-off, and audit trails

Hooks answer "did it pass the checks?", never "who is answerable?" The loop manufactures analysis, not accountability or legal standing.

  • Escalation triggers (stated in the goal file, not improvised): materiality threshold crossed, low confidence / out-of-range inputs, agents that cannot reconcile, a hook firing repeatedly, irreversible action. Human-review-required is a terminal state.
  • Sign-off (SR 11-7) = an event with time, identity, scope — not a property of the artifact. The skeptic is internal challenge, not independent validation. Carry an unsigned sign-off block the loop cannot complete.
  • Audit trail: version control + structured run log + content-addressed inputs ⇒ an auditor can regenerate every number. "Reproducible" means the derivation is recoverable and therefore challengeable.
What breaks, and what catches it

Six failure modes — each is omitted machinery

the meta-point Every control is a mechanism introduced earlier. The failure modes are the predictable consequences of omitting machinery the lecture already argued for — build the loop right and you have defended against most of them by construction.
Designs to reuse

Five design patterns — pick by the goal's properties

  • Planner–executor–critic — planning + execution + independent evaluation; for tasks where the next step depends on earlier results.
  • Generate–test–revise — the tightest loop; default when the goal is "make this check pass." Only as good as its test.
  • Scaffold–draft–audit–repair — for subjective, rubric-scored goals. This is the pattern behind this very book.
  • Data-pipeline: build–validate–document — the artifact is a process; lineage must be trustworthy.
  • Credit-model: build–stress-test–red-team–finalize — Example 2 generalized; stress-test and red-team are mandatory stages (SR 11-7 governance encoded as a pattern).
choosing — and composing Pick by the goal's properties (machine-checkable vs. judgment; number vs. artifact; stakes; reproducibility). Real workflows nest patterns. What never changes: the feedback must be informative.
Wrap-up

Lecture 17 in one idea — and what's next

The shift: from issuing instructions and accepting the reply, to designing a goal-directed loop whose output is observed, evaluated, and revised against an explicit objective.

  • Three building blocks: agents specialize the loop (who), skills reuse it (how), hooks enforce it (what it may never violate).
  • Three controls: goals (the setpoint), validators (honest feedback), stopping rules (terminate safely).

Appendix · lineage

The agentic-AI lineage — the loop is not new

LLM side — the loop generalizes a family of methods
  • ReAct (Yao 2022): interleave reason + act = Observe–Propose–Act
  • Reflexion (Shinn 2023): verbalized Evaluate–Revise
  • Self-Refine (Madaan 2023): single-model generate–critique–revise
  • Chain-of-thought (Wei 2022); Toolformer (Schick 2023); AutoGen (Wu 2023): split phases across role-typed agents
Engineering side — the loop predates LLMs by decades
  • Boehm's spiral model (1988): risk-driven cycles replace waterfall
  • Deming's PDCA: Plan–Do–Check–Act
  • The convergence is no coincidence: it is what any system producing correct output under uncertainty looks like once it closes its feedback loop.
One-shot = open-loop; goal-directed = feedback control. Error \(e_{t+1}=G-s_{t+1}\); instability when gain is too high (oscillation); steady-state error when the success predicate is biased.

Appendix · the enforcing hook

The root-finding stop hook — what makes "success" un-fakeable

The goal says what convergence means; the skill says how to measure it; this hook enforces it.

#!/usr/bin/env bash
# stop-hook: veto "success" unless the latest |f(x)| < 0.001
last=$(grep -o 'f(x)=[-0-9.eE]*' results.log | tail -n1 | cut -d= -f2)
abs=$(python -c "print(abs(float('$last')))")
ok=$(python -c "print(1 if abs(float('$last')) < 0.001 else 0)")
if [ "$ok" -ne 1 ]; then
  echo "BLOCKED: |f(x)|=$abs >= 0.001 -- not converged" >&2
  exit 1
fi
echo "OK: |f(x)|=$abs < 0.001"
exit 0
why it works The harness treats a nonzero exit from a stop hook as a veto: the agent literally cannot end the loop claiming success while the residual is too large.
