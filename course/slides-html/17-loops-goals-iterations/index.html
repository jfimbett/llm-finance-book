01

From linear prompts to iterative systems

The prompt is not the unit of design — the loop is.
Why the old way breaks

Why one-shot prompting is fragile

A single prompt is an open-loop controller: its output is consumed downstream with no channel to correct an error. The model never learns whether it was right.

  1. Errors propagate un-attenuated. One forward pass to parse, extract, compute, and narrate ⇒ an early slip corrupts everything after it, unnoticed.
  2. The spec exceeds one pass. Interacting constraints — turnover, sector caps, tracking error, tax lots — are satisfied jointly, and joint satisfaction falls roughly multiplicatively as constraints are added.
  3. The model cannot see the environment. It guesses the schema, the rate, whether a file exists — acting on a hallucinated world.
the bigger picture No analyst writes a DCF top-to-bottom and submits it. They sketch, sanity-check the terminal value, walk back an over-ceiling margin, re-run, circulate. Expert workflow is intrinsically iterative; one-shot prompting asks the model to do something no competent human would attempt.
The core mechanism

The goal-directed loop: five phases that repeat

Observe → Propose → Act → Evaluate → Revise, repeating until the goal holds or a stopping rule fires. The loop — not the prompt — is the primary unit of design.

  • Observe the state (files, tests, data, prior outputs)
  • Propose an action toward the goal, given that state
  • Act: execute it ⇒ a new state
  • Evaluate: run checks; does the success test hold?
  • Revise: fold the feedback back in, return to Observe
State \(s_t\); goal \(G\) with success predicate \(\mathrm{ok}(\cdot)\). Each step: \[ a_t=\pi(s_t),\quad s_{t+1}=\mathrm{Act}(a_t),\quad e_{t+1}=\mathrm{Eval}(s_{t+1}). \] Revise feeds the concrete evidence \(e_{t+1}\) of what went wrong back into the next Propose — the error-correction channel one-shot prompting lacks.
the Evaluate phase carries most of the value A loop is only as good as its feedback. A weak success test converges happily to something useless. Design the feedback with as much care as the prompt.
A crucial distinction

Looping vs. just repeating — genuine iteration carries information forward

Genuine iteration uses the feedback from one step to change the next action. Repetition without that feedback dependency is just sampling from the same distribution again.

Genuine iteration: \(a_{t+1}=\pi(s_{t+1},e_{t+1},h_t)\) — the next action depends on the evaluated feedback \(e_{t+1}\). If \(a_{t+1}\perp e_{t+1}\) given the original goal, the procedure is mere repetition (sampling), not iteration.
anti-pattern — spurious repetition Re-prompt "fix the failing test" without reading whether it now passes. Same number of model calls; no information carried forward; partial edits may leave the code worse than before.

Loop only if: output is mechanically verifiable; reasoning varies with input; the first attempt is likely wrong and costly; no irreversible harm before a human gate.

The concrete substrate

Claude Code as the loop substrate — each phase mapped to a primitive

The loop is an abstraction; Claude Code provides a concrete substrate where each of the five phases is a first-class, scriptable mechanism.

  • Observe → file-reading and command-running tools: the system reads the repository, queries a database, or reads a prior report before proposing anything.
  • Propose + Act → agents: role-scoped model invocations that call tools, edit files, and run commands.
  • Evaluate → whatever produces trustworthy feedback: a test suite, a validation script that exits nonzero on failure, a rubric applied by a separate "critic" agent, or a human review gate.
  • Skills package a recurring multi-phase workflow into a single named procedure, so an entire observe–propose–act–evaluate sequence can be invoked as a unit and reused (/score-content, /refine-until-threshold).
  • Hooks bind checks to lifecycle events — linter before a write, validator before a commit — so evaluation is enforced automatically.
  • Version-control commits are the loop's memory and rollback path: a passing iteration is committed; a bad iteration can be reverted.
why a concrete substrate matters The loop's reliability lives in its plumbing, not its prose. A success predicate that is "a human eyeballs it" is not durable. One that is "pytest exits zero and the rubric agent returns ≥ 90" is — because the substrate makes the Evaluate phase actually fire every iteration.
02

Goals as control objects

The setpoint of the loop — concrete, checkable, decomposable, and written down.
What a good goal is made of

Concrete, checkable, decomposable — one property per loop phase

The goal is the loop's setpoint: it governs behavior, decides when to stop, and decides what is acceptable. A vague goal yields a loop that wanders; a precise goal yields a loop that converges.

  • Concrete (for Act) — names the artifact and what to measure. "Improve the model" → "raise OOS AUC of credit.py, measured by backtest.py, to \(\geq 0.78\)."
  • Checkable (for Evaluate) — supplies the predicate \(\mathrm{ok}(\cdot)\): "every rubric dimension \(\geq 90\) via /score-content." Without it the loop has no way to know it is done.
  • Decomposable (for Revise) — splits into independently checkable subgoals; localizes failures and bounds context. Enables different subgoals to go to different specialized agents.
a complete goal specifies three distinct things Success criterion (stop because we won) · stopping rule (stop anyway: budget cap, no-progress detector) · anticipated failure modes (named in advance, each becomes a check). The common defect in practice: specifying only the first.
Three-component example

Success criterion, stopping rule, and anticipated failure modes — all three required

A goal to improve a credit model is complete only when it states all three components. Naming only the AUC target invites all three pathologies.

Success criterion
Out-of-sample AUC \(\geq 0.78\) on the held-out 2023 book, with calibration error below a stated bound.
Stopping rule
Halt after eight iterations, or when three consecutive iterations fail to improve validation AUC by more than \(0.002\).
Anticipated failure modes (each becomes a check)
  • Overfit to validation split → final lockbox test set evaluated once only.
  • Feature leakage (future information) → temporal-split check enforced by hook.
  • Calibration traded for discrimination → explicit calibration bound in success criterion.
the project's own controls max_refine_iterations in TOPIC.md is a stopping rule; quality_threshold is a success criterion. These are not settings — they are the two halves of a complete goal.
Make the goal durable

Write the goal as a file — ideally executable

A goal in a chat prompt vanishes when context rolls over. A goal in a file is durable, auditable, version-controlled — and, when executable, usable directly as the Evaluate phase.

# TASK: Harden the FCF valuation script
## Artifact
valuation/dcf.py  (value_company(ticker) -> float)
## Success criteria
- pytest tests/test_dcf.py exits 0
- Coverage of dcf.py >= 90%
- No division-by-zero on the 12 edge-case tickers
## Stopping rule
- Halt after 6 iterations, or 2 with no new passing tests.
## Anticipated failure modes
- Passing tests by catching all exceptions and returning 0
  -> forbidden; every value finite & positive or raise ValueError.
the decisive move Make "done" and "acceptable" the same code — a validate.sh that exits 0 iff the goal is met, wired into Evaluate and enforced before commit. The loop then becomes mechanical: act, run the script, read exit code, revise.

03

Agents, skills, and hooks

The who, the how, and the guardrails that don't depend on the agent.
The who

Agents are specialized iterators — five canonical roles

Useful agents are not general chatbots. Each occupies a narrow position inside the loop, performs one well-defined transformation, and hands control onward.

  • Worker — produces/transforms the primary artifact (the only mandatory role)
  • Critic — structured judgment without rewriting; the natural-language gradient that tells the worker in which direction to change
  • Validator — objective deterministic pass/fail; most often omitted, most sorely missed
  • Planner — decomposes a goal into ordered subtasks; reasons about sequence
  • Refactorer — improves internal structure while preserving observable behavior
critic vs. validator — a finance analogy The critic is the senior reviewer who says the model looks aggressive; the validator is the limit system that simply rejects the trade. A loop of only workers + critics can drift confidently in a wrong direction. The validator's verdict, built on a compiler or a test suite, is not subject to persuasion.
central discipline An agent's value is the durable change to state — files written, tests turned green, a commit recorded — not the text it prints. Measure it by its diff.
One agent or many?

Single-agent vs. multi-agent loops — when separation earns its cost

A single agent can play multiple roles in sequence (generate, then critique its own output, then revise). Separation earns its keep when the critic must be genuinely independent of the worker.

Single-agent self-reflection (Reflexion, Shinn 2023)
The agent reflects on the outcome of its own attempt, writes that reflection into memory, and conditions its next attempt on it. Simple to build; sufficient for many tasks.
Multi-agent separation (AutoGen, Wu 2023)
A model asked to critique its own freshly written text is biased toward endorsing it. Splitting into a separate critic agent, with its own prompt and no access to the worker's private reasoning, restores adversarial distance. The planner–executor–critic decomposition makes this explicit.
separation is not free Every additional agent adds orchestration cost, latency, and an opportunity for a lossy handoff. The right default: the fewest agents that give the critic real independence from the worker. Reach for a second agent when self-evaluation is untrustworthy, not when the diagram looks more sophisticated.
State between agents

Handoffs and shared state — the loop lives or dies on this

When more than one agent participates, the loop's reliability is determined by the quality of the handoffs. A lossy handoff is the most common multi-agent failure.

Channels — least to most fragile
  1. Durable project state — worker writes file, commits it; critic reads that committed file. Most robust.
  2. Structured output — a JSON object or scored report with a fixed schema passed between agents. Reliable if schema is stable.
  3. Free-form summary — prose summarized from one agent's context and pasted into another's. Most fragile.
the lossy-handoff failure A critic that receives only a prose summary of a chapter, rather than the chapter file, critiques the summary. Its change requests may be irrelevant to what is actually on the page. This is why this project routes handoffs through files and fixed-schema reports, not through conversational summaries — the scorer writes a JSON report to docs/quality/ with a stable schema, and the next agent reads that file.
trade-lifecycle analogy A trade economically agreed on the desk but transcribed incompletely into the settlement system produces a break that surfaces days later. The remedy is a single authoritative record — the booked trade; the committed file — that every downstream function reads from.
The how

Skills are reusable procedures — a skill is to a workflow what a function is to a computation

A skill is a named, composable sequence of steps that accomplishes a recurring task the same way every time (/draft-chapter, /score-content, /iterate-book-quality).

The skill–function correspondence
A name you invoke it by · scoped inputs (not ambient state) · a single responsibility · repeatable (same inputs → same procedure) · composable: /draft-chapter calls /score-content calls /refine-until-threshold.
  • Operational knowledge: "how we do X here" is written down once instead of re-derived from scratch in every session. The skill is the answer to "how do I add a chapter?" made executable.
  • Freeze conventions: /new-topic creates chapter/lecture/notebook N together — the numbering convention cannot drift because the only sanctioned way to create a topic enforces it.
  • Reduce entropy: every invocation pushes a long-running project back toward consistency. The hundredth chapter added through a skill has the same structure as the first. The skill library is the project's standard operating procedures — the control framework of an agentic project (Sarbanes–Oxley analogy).
The guardrail

Hooks: guardrails that don't depend on the agent's cooperation

A hook runs automatically on an event — pre-tool-use, post-write, session-end. Its defining property: it fires whether or not the agent remembers to call it.

Pre-action = pre-trade limit check (can block).  Post-action = post-trade surveillance (validate / record / repair / escalate).

The hook catalogue — five recurring categories
  • Pre-run checks: validate preconditions before a job starts (dataset checksum, no future-data leak).
  • Post-write validation: inspect the artifact the moment it is written (every \cite{} key resolves).
  • Linting, tests, compilation: mechanical correctness verdict after each edit (unit tests, LaTeX compile).
  • Safety checks: scan staged diffs for API keys, PII, or compliance violations before commit.
  • Commit checks: message format, quality gate green, integrity checks before a commit lands.
#!/bin/bash
# check-citations.sh — post-write validation hook
BIB="book/bibliography.bib"; MISSING=0
for f in $(git diff --cached --name-only | grep '\.tex$'); do
  for key in $(grep -oE '\\cite\{[^}]+\}' "$f" \
               | sed -E 's/\\cite\{([^}]+)\}/\1/'); do
    grep -q "{$key," "$BIB" || { echo "Unresolved: $key"; MISSING=$((MISSING+1)); }
  done
done
[ "$MISSING" -gt 0 ] && { echo "FAIL"; exit 1; }; echo "OK"; exit 0
Making feedback trustworthy

A passing check is necessary, not sufficient — Goodhart's law

The loop optimizes for the check, not the goal. If the check is easy to pass without meeting the goal, an optimizing loop will find the gap.

Let \(V\) be the validator. The loop optimizes for \(V=\text{pass}\), not the true goal \(G\). Unless \[ \{x: V(x)=\text{pass}\}\subseteq\{x: G\text{ met}\}, \] there exist artifacts that pass yet fail the goal — and an optimizing loop will find them. (Goodhart's law, 1984: when a measure becomes a target, it ceases to be a good measure.)

Make \(V\) hard to satisfy without meeting \(G\):

  • Outcome-tied, not proxy: testing that a backtest is leak-free and returns reconcile is harder to game than asserting the word "Sharpe" appears.
  • Diverse checks: gaming the suite requires gaming all of them simultaneously.
  • Validator logic out of the worker's reach — it cannot tailor output to the test's internals.
  • Hidden holdout at release, so the loop cannot overfit to the full battery it sees during iteration.
Failure hooks — what fires on a red verdict (escalating in severity)
Block (refuse) · Revert (undo to last-good) · Escalate (hand to a human) · Re-enter (feed the failure back as next input — the default when fixable).

Designing an iteration loop

A seven-step template, its pseudocode, state on disk, stopping rules, and economics.
The template

Seven steps — and three of them must stay separate

The same loop structure appears in Boehm's spiral model (1988), Deming's Plan–Do–Check–Act cycle (1986), and every agentic system that produces correct output under uncertainty.

  1. Read state — ground truth from disk + version control first
  2. State the next local goal — a checkable target for this turn
  3. Propose a change (reason)
  4. Apply it (act — write to disk; only now has the world changed)
  5. Validate — a deterministic check; the agent does not grade its own work
  6. Interpret feedback — direction and magnitude of the error
  7. Decide: stop / continue / escalate
two non-negotiables Validation (5), interpretation (6), and decision (7) are separate — collapsing them removes the independent check. And the decision always includes escalate: a loop that can only succeed or grind forever will grind forever.
From template to code

Pseudocode — mapped onto Claude Code primitives

Every line corresponds to a primitive you already know: file reads, a checked-in goal, a skill that validates, a commit that records.

state <- read_state()            # file reads (agents, git log, last report)
goal  <- read_goal()             # checked-in goal file
best  <- evaluate(state, goal)
while not goal.satisfied(best) and not goal.budget_exhausted():
    local  <- choose_objective(state, best)
    change <- propose(state, local)     # REASON (ReAct)
    state2 <- apply(change, state)      # ACT: write to disk
    report <- validate(state2)          # a SKILL: deterministic check
    if report.improves_on(best):
        commit(state2, summarize(change, report))   # audit trail
        state, best <- state2, report.score         # accept (Self-Refine)
    else:
        state <- rollback(state)                     # reject
    if stalled(history) or report.requires_human:
        escalate(state, report); break               # hook-gated
emit_report(state, best, history)                     # final summary
why this is robust The accept/reject discipline makes the loop robust to a noisy proposer: a bad proposal costs one iteration, not the whole run — because validation and version control are deterministic.
Memory and state

State lives on disk — design for amnesia

design for amnesia The agent is a new employee every morning who reads the project's files, does one day's work, commits it, and is replaced overnight by someone equally capable but with no recollection of yesterday. Write before you forget.
Persists (external to the model)
Files in working tree · git history (long-term memory) · validation reports at known paths · a status scratchpad (current best, iterations spent, what failed). Bonus: all-on-disk state makes the loop resumable after a crash or budget pause.
lost every turn Chain-of-thought from prior turns · intermediate values not saved · nuance of earlier reasoning not committed. Practical rule: a loop whose correctness depends on the agent remembering the last guess is broken.
When to halt

Stopping criteria — exactly one must be guaranteed-terminating

A loop that cannot stop is a liability. The loop must carry at least one criterion that fires regardless of whether the goal is met.

A stopping rule is a predicate over the loop's history that returns true when the loop must halt. A loop is well-formed only if at least one of its stopping rules is guaranteed-terminating: it fires after finitely many iterations independent of progress.
Four stopping rules — one must be guaranteed-terminating
  • Success — validator reports the success predicate holds. Desired exit; not guaranteed.
  • Convergence — best score unchanged for \(k\) consecutive iterations. Loop is stuck at a local optimum.
  • Budget exhaustion — max iteration count, wall-clock, or token budget. This is the mandatory guaranteed-terminating criterion.
  • Human-review-required — a guardrail flagged a condition policy forbids resolving automatically.

Convergence and human-review make the loop smart about stopping early; the budget makes it safe. Convergence/success fire in the common case; budget fires only in pathological ones — but it is always present.

Cost and value

The economics of iteration — stop where marginal value crosses marginal cost

Each iteration has a cost. As the loop approaches the goal the marginal improvement per iteration shrinks while cost stays flat. There is a point past which the next iteration costs more than the improvement it delivers.

cost–value sketch (hypothetical numbers) Analyst manual revision: 2 hours at \(\$\)200. Each agentic iteration: ≈ \(\$\)0.50 in tokens plus 90 s. Eight iterations reach acceptable quality at ≈ \(\$\)4 and 4 minutes — an obvious win. Iterations 15–30 might cost another \(\$\)8 to chase quality a reviewer cannot distinguish from the eighth. Stop at the crossing point, by rule. (All figures illustrative.)
Interactive setting
Wall-clock time is the binding budget. A loop converging in 30 cheap iterations but taking 40 minutes may be unusable even if its dollar cost is trivial.
Overnight / batch setting
Dollar and token cost bind instead. Name which budget is binding at design time; the stopping rule should encode that business decision.

Example 1: derivative-free root finding

The whole machinery, stripped bare — a loop that improves on feedback alone.
The stripped-bare loop

A loop that improves using feedback alone — no calculus allowed

Find an \(x\) where a function is (nearly) zero, using only the sign and size of the residual. The pedagogical point is not numerical analysis — it is that the same loop machinery works when the residual is a failing test or a reviewer's objection.

Find \(x\) with \(f(x)=x^3-x-2\approx 0\) to tolerance \(|f(x)|<10^{-3}\) (single root near \(x\approx 1.52\)). The agent may evaluate \(f\) anywhere but is forbidden derivatives — no \(f'(x)=3x^2-1\), no Newton update. It reads only the sign and size of the residual.
finance analogue: IRR and implied volatility To find an IRR, seek rate \(r\) with \(\mathrm{NPV}(r)=\sum_t c_t/(1+r)^t=0\) — no closed form. Evaluate at trial rates and move by the sign: too-high NPV means rate is too low. Implied volatility works the same way. Swap \(f\), the loop is unchanged.

Three artifacts, none containing an algorithm: a goal file (\(|f|<0.001\), budget ≤ 12 iters), a skill (python evaluator.py $x | tee -a results.log), and a stop hook that vetoes "success" unless the last \(|f(x)|<10^{-3}\).

Watching it converge

Iteration trace from a deliberately poor start (x = 0)

Iter\(x\)\(f(x)\)Agent's reasoning
1\(0\)\(-2\)Negative; root to the right. Jump up.
2\(2\)\(4\)Positive; root bracketed in \((0,2)\).
3\(1.5\)\(-0.125\)Negative but small; root just above \(1.5\).
4\(1.6\)\(0.496\)Positive; bracket \((1.5,1.6)\).

The sign change between \(0\) and \(2\) is the engine: a continuous function going from negative to positive crosses zero between them. The magnitudes place the root near \(1.5\) (false-position reading). Declaring success at iteration 5 would be vetoed by the stop hook.

why the inefficiency is the point The loop reached tolerance without an algorithm, using only feedback — so the same loop works when no update formula exists at all.
The final report

What the loop emits when it stops — reconstructable from disk alone

The emit_report step produces a human-readable summary an auditor can consume in one pass — reconstructed entirely from results.log and the git history, never from the agent's memory.

this is the template every later chapter instantiates Only the state variable, the validator, and the definition of "residual" change. The goal file states what was asked; the log holds the full trace; the hook's exit code certifies convergence; the git history orders the accepted states.

Example 2: credit-risk stress-testing loop

From a point estimate to a memo a committee would actually accept.
The goal

A defensible credit-committee memo — not a point estimate

Meridian Components (fictional; all figures illustrative): a mid-market manufacturer seeking a \(\$\)40m revolver. Looks comfortable — growth, rising EBITDA, leverage in appetite — but cyclical end markets, illiquid specialized inventory, concentrated receivables (top 2 customers = 45% of receivables), and a cash conversion cycle that crept from 70 to 95 days.

the goal is not "estimate the PD" A point estimate cannot be wrong — nothing fails against it. The goal is a memo a committee would accept: that acceptance is checkable, so the loop runs until the checks pass. Credit scoring has a long quantitative tradition (Thomas, 2002); any PD model used for lending is a model-risk artifact subject to SR 11-7 governance.
\[ \mathrm{EL} = \mathrm{PD}\times\mathrm{LGD}\times\mathrm{EAD}. \] PD = 1-yr default probability; \(\mathrm{LGD}=1-\text{recovery rate}\); EAD = drawn balance plus estimated drawdown. This identity must reconcile in every column of the memo.

The memo must give PD, LGD, EAD, EL (dollars and bps), risk drivers, and a recommendation — lend / modify / decline. A "low risk" phrase is gated on both base and stressed EL clearing explicit thresholds.

The shock

Scenario layering: base, adverse, severe — combined and correlated

Sound stress testing uses coherent combined scenarios, not one-at-a-time bumps. Correlated drivers (revenue shock, rate rise, slower collections) trigger breaches only when they coincide.

Base case: the central expectation. Adverse: a plausible but unfavorable combined shock. Severe: stresses the same drivers further, into the tail, to test whether the structure holds when risks correlate and arrive together. (BCBS 2018 stress-testing principles.)
Shock parameterAdverseSevere
Revenue\(-20\%\)\(-30\%\)

Benchmark rate\(+250\) bps\(+350\) bps

Combined because the parameters are correlated: the same macro event cuts revenue, widens credit spreads, and slows collections simultaneously. Running one-at-a-time understates the joint tail and misses covenant breaches that only trigger when several stresses coincide.

The ensemble

Nine role-scoped agents — two feedback edges turn a pipeline into a loop

Nine agents (each output → next input)
  1. Data-ingestion
  2. Accounting-quality (normalize; strips add-backs)
  3. Feature-engineering (leverage, coverage, CCC, concentration)
  4. PD-model (leakage hook can reject)
  5. Stress-scenario (applies parameter table)
  6. Cash-flow (covenants, liquidity)
  7. LGD (collateral valuation, recovery)
  8. Skeptic — returns to ANY agent above
  9. Memo-writing (skeptic-gated)

Two feedback edges (skeptic + failing hooks) turn a pipeline into a loop (AutoGen pattern; Wu 2023).

the two load-bearing hooks Leakage check: the easy path to a beautiful backtest is to let the model peek at the answer — this hook blocks it before any PD is reported.

"Low risk" gate: language is a control surface — the phrase must be earned by the arithmetic, not chosen by the writer.
The skills

Seven named skills — freezing "how we do it" into an auditable procedure

Skills freeze the procedure so that the same normalization, the same feature definitions, and the same modeling discipline are applied to this borrower and the next. A skill captures "how we do X here" in executable, version-controlled form.

  • financial-statement-normalization — standardizes line items, strips non-recurring add-backs, enforces one consistent working-capital definition.
  • credit-feature-engineering — computes the credit feature set from normalized statements with documented, versioned formulas.
  • pd-modeling — train/test split before fitting; leakage checks; OOS validation; reject and stop if leakage detected or calibration poor.
  • macro-stress-scenario-design — expands the parameter table into internally consistent base/adverse/severe input sets.
  • covenant-liquidity-analysis — projects coverage ratios, headroom, and minimum-liquidity compliance per scenario.
  • lgd-collateral-valuation — values collateral, applies haircuts, derives recovery and LGD by scenario.
  • credit-memo-writing — assembles validated artifacts into the committee memo in the required structure.
why skills matter in regulated credit Because the procedure is captured once, it is auditable and repeatable. An agent cannot quietly skip the leakage check or the OOS validation — both are written into the pd-modeling skill and enforced by the corresponding hook from outside.
The hooks

Eight automatic checks — converting "is this sound?" into a pass/fail gate

In a regulated credit process, hooks are the difference between a memo and a liability. They convert the soft question "is this analysis sound?" into automatic, enforceable gates.

  1. No uncited factual claims — every assertion points to a source line item.
  2. Ratios reconcile — every ratio recomputes from normalized statements within tolerance.
  3. Train/test separation — holdout carved before any fitting; overlap aborts the run.
  4. Target-leakage check — no feature may encode the default outcome; positive signal rejects the model.
  1. Parameterized scenarios — every scenario ships with a parameter table; prose-only rejected.
  2. Skeptic sign-off — recommendation not final until skeptic has explicitly signed off.
  3. End-to-end regeneration — all tables/figures must be regenerable from code in one pass.
  4. "Low risk" threshold gate — memo may not use "low risk" unless base and stressed EL are both below explicit thresholds.
Goodhart + skeptic as the defense Hooks catch the violations you can specify in advance. The skeptic (a Reflexion-style critic; Shinn 2023) catches the ones you cannot — it demands the analyses that would falsify the conclusion and feeds that critique back into the next iteration.

Human oversight, accountability, and audit trails

What a signature adds that a hook cannot — and how to make the trail reconstructable.
Accountability, not just analysis

Human-in-the-loop — escalation triggers and the unsigned sign-off block

Hooks answer "did it pass the checks?", never "who is answerable?" The loop manufactures analysis, not accountability or legal standing.

An escalation trigger is a predicate, evaluated automatically at the end of an iteration, that forces the loop into the human-review-required state when it is true. Unlike a hook (which blocks a single bad artifact), an escalation trigger blocks the loop's authority to proceed and transfers the decision to a named human.
  • Stated in advance in the goal file: materiality threshold crossed (e.g. facility above a hypothetical \(\$\)25m limit) · low model confidence or out-of-range inputs · agents cannot reconcile · a hook fires repeatedly (loop stuck) · proposed action is irreversible.
  • The unsigned sign-off block: the artifact carries an explicit block only a human can complete. The loop fills in everything except the signature — preventing the system from simulating sign-off.
a loop with no escalation triggers has quietly assumed that nothing it does is consequential. A useful first question when reviewing an agentic financial system: "under what conditions was it required to stop and ask a human?"
SR 11-7 governance

Accountability and sign-off — the skeptic is not independent validation

Supervisory guidance on model risk (SR 11-7) organizes governance around three pillars: sound development, independent validation, and active governance with clear ownership and accountability. Automating construction with an agentic loop dissolves none of these pillars.

Separation of roles
The party that develops a model must not be the sole party that validates it. The worker agents are developers; the skeptic agent plays the role of challenge — but an internal challenger implemented by the same team that wrote the workers is not independent validation in the regulatory sense.
Model ownership and sign-off
A model is owned by a named individual or committee accountable for its use and consequences. Sign-off is the recorded act by which that owner and an independent validator attest the model has been developed, tested, and validated to standard and is approved for a specific use. Sign-off is an event with a time, an identity, and a scope — not a property of the artifact.
what the loop can and cannot do The skeptic's approval is a precondition that makes the memo eligible for human review. The sign-off is the credit officer's recorded decision. The loop makes sign-off easy and well-evidenced — it must never simulate it. Stress testing as a governance tool, owned by senior management (BCBS 2018), is realized only when a human treats it as a reason to challenge the lend decision.
Reproducibility

Audit trails — three records that make derivations challengeable

The standard for a high-stakes financial loop: anyone not present should be able to reconstruct exactly how every number was produced, months later.

Three records that together suffice
  • Version control: every change is a commit with author, timestamp, message.
  • Structured run log: each iteration appends a machine-readable record of which agent ran, which skill, which hooks fired, and the artifact's measured state afterward.
  • Content-addressed inputs: data and model weights referenced by hash — "the data" is an unambiguous object.

Goodhart in depth

Overfitting to validation checks — the mechanism and the remedies

When a loop optimizes against a fixed set of checks, it will find ways to satisfy the checks that do not serve the underlying purpose. The agent is not malicious — it is doing what the loop rewards.

Let a loop optimize an artifact against a fixed validator \(V\) as a proxy for true objective \(U\). As optimization pressure increases, the correlation between \(V\) and \(U\) on the optimized artifacts weakens: the artifacts that maximize \(V\) are increasingly those that exploit the gap between \(V\) and \(U\) rather than those that achieve \(U\). A model agent rewarded for AUC will reach for leakage; a memo agent rewarded for a clean checklist will phrase around it.
the finance analogy is direct A desk managed to a single regulatory capital ratio learns to optimize that ratio while shifting risk into corners the ratio does not measure. An agentic loop optimizing against a single compliance check will reinvent this behavior automatically — with no intent and at high speed.
  • Diverse checks: a battery of heterogeneous validators is harder to game simultaneously.
  • Held-out validation data: the loop never sees it during optimization.
  • Adversarial review: a critic whose explicit job is to find how the artifact games the checks — rotated or re-specified so the loop cannot overfit to the critic either.
Designs to reuse

Five design patterns — pick by the goal's properties

  • Planner–executor–critic — planning + execution + independent evaluation; for tasks where the next step depends on earlier results. (ReAct + Reflexion.)
  • Generate–test–revise — the tightest loop; default when the goal is "make this check pass." Only as good as its test; natural breeding ground for Goodhart overfitting.
  • Scaffold–draft–audit–repair — for subjective, rubric-scored goals. This is the pattern behind this very book. The audit is itself a judgment.
  • Data-pipeline: build–validate–document — the artifact is a process; lineage must be trustworthy. Document step is a first-class output enforced by a hook.
  • Credit-model: build–stress-test–red-team–finalize — Example 2 generalized; stress-test and red-team are mandatory stages (SR 11-7 governance encoded as a pattern).
choosing Pick by the goal's properties: machine-checkable vs. judgment; single number vs. complex artifact; stakes; reproducibility requirement. What never changes: the feedback must be informative.

Wrap-up

Lecture 17 in one idea — and what's next

The shift: from issuing instructions and accepting the reply, to designing a goal-directed loop whose output is observed, evaluated, and revised against an explicit objective.

  • Three building blocks: agents specialize the loop (who), skills reuse it (how), hooks enforce it (what it may never violate).
  • Three controls: goals (the setpoint), validators (honest feedback), stopping rules (terminate safely).
  • Finance demands: explicit goals, real and independent validation, mandatory stress testing, and built-in skepticism — because a loop optimizing against fixed checks will, left alone, learn to game them.

Appendix · lineage

The agentic-AI lineage — the loop is not new

LLM side — the loop generalizes a family of methods
  • ReAct (Yao, 2022): interleave reason + act = Observe–Propose–Act
  • Reflexion (Shinn, 2023): verbalized Evaluate–Revise; single-model self-reflection
  • Self-Refine (Madaan, 2023): single-model generate–critique–revise
  • Chain-of-thought (Wei, 2022): makes Propose phase's reasoning explicit
  • Toolformer (Schick, 2023): gives Act phase real effects in the world
  • AutoGen (Wu, 2023): split phases across role-typed agents
  • Generative Agents (Park, 2023): long-horizon, memory-bearing behavior
Engineering side — the loop predates LLMs by decades
  • Boehm's spiral model (1988): risk-driven cycles replace waterfall
  • Deming's PDCA (1986): Plan–Do–Check–Act; same five phases in four
  • The convergence is no coincidence: it is what any system producing correct output under uncertainty looks like once it closes its feedback loop.
One-shot = open-loop control; goal-directed = feedback control. Error \(e_{t+1}=G-s_{t+1}\); instability when gain is too high (oscillation); steady-state error when the success predicate is biased. Predicts the failure modes of §8 from first principles.

Appendix · the enforcing hook

The root-finding stop hook — what makes "success" un-fakeable

The goal says what convergence means; the skill says how to measure it; this hook enforces it — and the agent literally cannot end the loop while the residual is too large.

#!/usr/bin/env bash
# stop-hook: veto "success" unless the latest |f(x)| < 0.001
last=$(grep -o 'f(x)=[-0-9.eE]*' results.log | tail -n1 | cut -d= -f2)
abs=$(python -c "print(abs(float('$last')))")
ok=$(python -c "print(1 if abs(float('$last')) < 0.001 else 0)")
if [ "$ok" -ne 1 ]; then
  echo "BLOCKED: |f(x)|=$abs >= 0.001 -- not converged" >&2
  exit 1
fi
echo "OK: |f(x)|=$abs < 0.001"
exit 0
why it works The harness treats a nonzero exit from a stop hook as a veto: the agent literally cannot end the loop claiming success while the residual is too large. This is the external validator of §4 made concrete.
