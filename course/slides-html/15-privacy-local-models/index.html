01

The privacy imperative in financial AI

Why privacy — not accuracy or cost — is often the binding constraint on deploying LLMs in a bank.
The founding cautionary tale

Why privacy is the binding constraint

January 2023 A major bank bans staff from pasting client data into ChatGPT after a leaked memo reveals loan applications, earnings transcripts, and merger term sheets crossing a public API. Weeks later, Samsung source code submitted to ChatGPT is retained for training. Once data crosses the API boundary, there is no technical mechanism for recall.

LLMs are useful because they absorbed vast text — but absorption plus cloud inference creates a privacy surface that finance is poorly equipped to govern.

Four complementary engineering strategies — the spine of this lecture
  1. Local / self-hosted deployment — remove the external boundary
  2. Text anonymisation — strip sensitive content before the model sees it
  3. Privacy-preserving training (DP, federated learning) — bound memorisation
  4. Privacy-aware architecture (RAG, audit, access control) — govern the system
Two ways data escapes

An LLM exposes data through two distinct channels

A model can betray information either through what it remembers from training, or through what you type into it. The fixes are different, so the channels matter.

Training channel
Information in the pre-training / fine-tuning corpus is reconstructable from the parameters.
  • Persists after the session ends
  • Any user may recover fragments
  • Fine-tune on proprietary docs ⇒ a permanent artefact
Inference channel
Data in prompts may be stored, logged, or trained on by the provider.
  • Operational; depends on the contract
  • Eliminated by on-premise deployment
  • Encountered first — no skill needed to exploit it
memorisation is real and quantifiable Carlini et al. (2021): a 1.5B-param GPT-2 memorises verbatim sequences — emails, phone numbers, identifiers seen once. Larger models memorise more; repeated sequences memorised most. Carlini et al. (2022) formalise a per-example memorisation score.
Making it precise

What does it mean for a model to "memorise" something?

In plain words: a model memorises a string if, given the right nudge, it will spit that exact string back out — not paraphrase it, reproduce it.

  • Exact memorisation: the string is reproduced verbatim.
  • Approximate memorisation: reproduced within a bounded number of single-character edits (insert / delete / substitute — the Levenshtein distance).
the intuition A client name appearing once in a fine-tuning corpus is memorised; the word "the" is learned. Rarity, in part, decides which.
A language model \(\mathcal{M}\) memorises a string \(s\) if there exists a prompt \(p\) such that \(\mathcal{M}(p)\) generates \(s\) with probability above a threshold \(\tau \in (0,1]\).

Why rare sequences get memorised: the next-token objective rewards generalisable features for common patterns, but for a rare pattern the only path to low loss is to store the specific sequence. The exact-vs-approximate distinction uses Levenshtein edit distance — the minimum number of single-character inserts, deletes, or substitutions.

02

Privacy risks in LLM deployment

Three mechanisms by which a model leaks — none of which a firewall or encryption can stop.
The threat map

Three risk mechanisms, unlike any database breach

the bigger picture Privacy risks live on the confidentiality axis of the CIA triad — but through mechanisms unlike a database breach. No firewall stops membership inference; no encryption stops an LLM from reciting a training example. Knowing the mechanism is a prerequisite for choosing the mitigation.
  1. Training-data memorisation & leakage — extraction attacks recover rare / repeated sequences from parameters.
  2. Inference attacks — membership and property inference leak information without verbatim memorisation.
  3. Prompt injection & exfiltration — adversarial instructions in retrieved / ingested content hijack the model.
Risk 1 · memorisation

How an attacker pulls training data back out of a model

Neural models compress training data into parameters — but the compression is lossy only in aggregate. Unusual, repeated, or long examples are retained with high fidelity, and can be fished back out.

  1. Generate many long continuations from varied prompts.
  2. Rank each candidate \(s\) by \(\dfrac{\Pr_{\text{target}}(s)}{\Pr_{\text{reference}}(s)}\) — the target model versus a smaller reference model.
  3. A high ratio \(\Rightarrow\) the string is disproportionately likely to be training data (the reference model has not seen the specific sequence).
Recovered in practice: verbatim emails, phone numbers, names, full PII-bearing examples.
Risk 1 · the finance twist

Why memorisation is worse in finance, not better

amplified in finance Fine-tuning on internal research / deal docs / client comms is attractive — and embeds those docs in the parameters. A competitor querying the model can recover fragments even behind an access-controlled API.

The extraction risk scales with how rare and how repeated a sequence is. Proprietary deal documents are unluckily both.

Why deal docs are the worst case
They appear verbatim across many drafts (repeated), and their exact phrasing exists nowhere in the public pre-training data (rare) — together maximising the likelihood ratio that exposes them.
Risk 2 · inference attacks

Leaking data without ever reciting it

An attacker need not extract a document verbatim. Just asking "was this in the training set?" — and getting a better-than-coin-flip answer — already leaks.

Financial consequence
A bank fine-tunes on merger targets; an adversary queries each candidate name and reconstructs the deal pipeline with above-chance accuracy. The same logic leaks whether a specific credit file was in training.
For a model \(\mathcal{M}\) trained on \(D\) and an adversary \(\mathcal{A}\) observing a candidate \(s\): \[ \mathrm{Adv}(\mathcal{A}) = \bigl|\Pr[\mathcal{A}(\mathcal{M}, s){=}1 \mid s \in D] - \Pr[\mathcal{A}(\mathcal{M}, s){=}1 \mid s \notin D]\bigr| \] \(0\) = no discrimination; \(1\) = perfect discrimination. The basis: models assign higher likelihood to their training examples.

Property inference extends this to aggregates (mean loan size, fraction delinquent) — harder to detect, and requires no example-level memorisation.

Risk 3 · prompt injection

When a document the model reads gives it orders

Prompt injection (Perez & Ribeiro, 2022): malicious instructions hidden in processed content override the system's own instructions. In RAG, retrieved documents get concatenated into the prompt — so whoever controls a document controls the instructions.

Two finance-specific exposures
  • Adverse-media screening — an attacker publishes an article with an injection payload ⇒ forces a favourable risk score, or leaks the screening criteria.
  • Counterparty document ingestion — loan applications, due-diligence questionnaires, ISDA agreements carrying embedded instructions.
no complete solution yet Best current mitigations: structural separation of trusted vs. untrusted content · output sanitisation · rule-based / constitutional classifiers scanning output before delivery.
03

Local and self-hosted deployments

The simplest fix for the inference channel: never let the data leave your network.
Strategy 1

Keep inference inside the perimeter

the bigger picture The simplest way to kill inference-time exfiltration is to keep inference on the institution's own infrastructure. For many financial workflows the mix of regulation, contracts, and competitive sensitivity makes on-premise not merely preferable but mandatory. Capable open-weight models (2023–2024) made this practically feasible for the first time.
"Open-weight" — the term that makes this possible
Parameters are publicly released; inference needs no third-party API call. (Distinct from fully open-source — weights yes, full training code / data often no.) What matters for privacy: no data crosses an external boundary.

Where to run it

On-premise vs. private cloud — and how to route between them

On-premise
  • + No data leaves the perimeter (prompts, completions, metadata)
  • – GPU capex; ops burden; less elastic
Private cloud
  • Single-tenant / VPC, dedicated hardware
  • AWS Outposts, Azure Gov, GCP Assured Workloads
  • ! DPA must satisfy GDPR Art. 28 & DORA Art. 30
routing for a commercial bank A lightweight classifier sorts each request:
  1. Public: no client IDs → cloud API
  2. Internal: internal analysis, no PII → private-cloud single-tenant
  3. Client: names / accounts / txns → on-prem, no external network
The classifier trains on internal data, calls no external API. Every routing decision is logged (timestamp + tier).
The price of staying local

You trade capability for control — and shrink the model to fit

The cost of going local is capability: the best open weights (2025) sit around GPT-3.5-class, lagging the frontier (GPT-5 / Claude Opus 4.8) on complex reasoning and broad factual recall. To make them fit your hardware, you quantise — store the weights at lower precision.

Memory at 4-bit
7B ≈ 3.5 GB (less than a video game) · 13B ≈ 6.5 GB — fits a high-end consumer GPU. A fine-tuned model can be quantised on-prem in < 1 hr.
Quantisation lowers weight precision (16/32-bit \(\to\) 4-bit).
  • GPTQ (Frantar et al., 2022): block-wise 2nd-order reconstruction using the Hessian; takes hours; needs a GPU.
  • AWQ (Lin et al., 2023): ~1% of weights are salient (large activations). Protect them with a per-channel scale; quantise the rest uniformly. Minutes; no GPU needed.
Similar final perplexity. The quant step uses only a small public calibration set — no client data, no added privacy risk.
04

Text anonymisation and de-identification

When the data must leave the building, strip the sensitive parts before the model sees them.
Strategy 2

When local deployment is not enough

the bigger picture When inference must pass through external systems, the alternative is to remove or obscure sensitive information before the text reaches the model. The technique — de-identification — comes from clinical informatics; finance swaps diagnoses for client names, account numbers, LEIs, and trading positions.
the challenge Sensitive data sits in prose, not labelled fields — a loan officer's narrative, call notes, an analyst's draft. Removing a name without context breaks intelligibility; replacing it must be consistent so co-references stay coherent.
The detection step

Named-entity recognition finds the sensitive spans

Modern NER = pre-trained transformer encoders, fine-tuned; F1 > 0.90 on standard entity types (person, org, location, date, money) in well-edited English. Finance adds entity types general NER misses.

  • Account IDs — IBAN, SWIFT / BIC, account numbers
  • Securities — ISIN (12-char), CUSIP, tickers
  • Legal entities — LEI (20-char), registration numbers
  • Regulatory — passport, national ID, tax ID
Production pipeline = two layers (spaCy; Honnibal & Montani 2017)
A learned model for prose entities (names, orgs, locations) plus a rule-based matcher for structured identifiers (IBAN, ISIN, LEI). Detected by regex or a fine-tuned NER layer.

\(\mathcal{D}\) satisfies \(k\)-anonymity on quasi-identifiers \(\mathcal{Q}\) if every record is indistinguishable from \(\geq k-1\) others when restricted to \(\mathcal{Q}\). For text: any output could plausibly come from \(\geq k\) individuals. Masking trivially gives large \(k\) — but surrounding context may still re-identify.
Making the masked text read naturally

Pseudonymisation, tokenisation, and the re-identification catch

Pseudonymisation
Replace each entity with a realistic fake surrogate ("Marie Leclerc" for "Antoine Dupont"), consistent across the document.
  • + Reads natural — the LLM does not detect placeholders and degrade to generic output
  • – The mapping table is a key: GDPR Recital 26 treats pseudonymised data as protected, not anonymised. Hold the key separately, access-controlled, encrypted at rest.
Format-preserving tokenisation
Replace with a same-format token (a valid-but-fake IBAN). Standard in PCI-DSS; now in LLM pipelines — the data looks real without being real.
contextual re-identification Even with all entities removed, a unique combination of dates, amount-ranges, and product types may match one client. Risk grows with the adversary's background knowledge \(\mathcal{K}\), which grows over time. ⇒ re-assess anonymisation adequacy on a periodic cycle, especially for long-lived archives.

Privacy-preserving training

Anonymisation guards the inference channel. These techniques guard the training channel.
Strategy 3

Bounding what the model permanently memorises

the bigger picture Anonymisation guards the inference channel. This section guards the training channel: limiting how much sensitive information becomes permanently embedded during fine-tuning. Two complementary techniques — differential privacy bounds per-example contribution; federated learning keeps data in place.
The gold standard

Differential privacy: a mathematical promise of deniability

The promise: an adversary who sees the model's output cannot confidently tell whether any single record was in the training set. Add the right amount of noise, and every individual gets plausible deniability.

  • \(\varepsilon \geq 0\) = the privacy budget — smaller means more private
  • \(\delta \geq 0\) = the failure probability — pure DP has \(\delta = 0\)
the intuition The log-odds of distinguishing two near-identical datasets are bounded by \(\varepsilon\) — exactly, with probability \(1-\delta\).
A randomised mechanism \(\mathcal{M}\) is \((\varepsilon,\delta)\)-DP if for all adjacent datasets \(D, D'\) (differing in one record) and all output sets \(S\): \[ \Pr[\mathcal{M}(D) \in S] \leq e^{\varepsilon}\,\Pr[\mathcal{M}(D') \in S] + \delta. \]
From definition to algorithm

How you actually train with differential privacy

The recipe is to bound how much any one example can change the model, then drown that bounded influence in calibrated noise. Applied to gradient descent, it is called DP-SGD.

For a function \(f\) with \(\ell_2\)-sensitivity \(\Delta_2 f = \max_{D,D'}\|f(D)-f(D')\|_2\): \[ \mathcal{M}(D) = f(D) + \mathcal{N}(0,\sigma^2 I_d),\quad \sigma = \frac{\Delta_2 f \cdot \sqrt{2\ln(1.25/\delta)}}{\varepsilon} \] is \((\varepsilon,\delta)\)-DP for \(\varepsilon,\delta \in (0,1)\).
  1. Per-example gradients \(g_i\) (not the minibatch sum).
  2. Clip: \(\hat{g}_i = g_i / \max(1, \|g_i\|_2/C)\) — bounds each example's sensitivity to \(C\).
  3. Noise: add \(\mathcal{N}(0, \sigma^2 C^2 I)\) to the clipped sum before the update.
The budget is tracked by the moments accountant / Rényi-DP — tighter than basic composition.
The bill comes due

Privacy costs accuracy — and every run spends from a finite budget

The cost of DP-SGD is utility: the noise slows convergence.

  • \(\varepsilon = 1\): strong privacy, larger utility cost
  • \(\varepsilon = 10\): weak privacy, small cost

Accuracy rises with \(\varepsilon\) and saturates toward the non-private baseline.

A useful pairing
LoRA / adapters pair well with DP-SGD: fewer trainable params ⇒ lower sensitivity ⇒ less noise.
Sequential composition: \((\varepsilon_1{+}\varepsilon_2,\,\delta_1{+}\delta_2)\).

Advanced composition over \(k\) steps: \[ \varepsilon' \approx \varepsilon\sqrt{2k\ln(1/\delta')}. \]
operational implication Every DP fine-tune spends budget. Plan cumulative loss across runs; once exhausted, further training violates the guarantee — requiring a fresh model or a new budget allocation.
When the data can't even be moved

Federated learning: train across silos without pooling the data

A different threat: data centralisation. Residency rules may forbid moving client data across a jurisdiction — even within one institution. Federated learning sends the model to the data, not the data to the model.

\(K\) clients hold \(D_k\) (\(n_k\) examples, \(n=\sum n_k\)). Minimise \(F(\theta)=\sum_k \frac{n_k}{n}F_k(\theta)\). Each round \(t\): selected clients init with \(\theta^t\), run \(E\) local steps, and return \(\Delta_k^t\). The server aggregates: \[ \theta^{t+1} = \theta^t + \sum_{k \in S_t}\frac{n_k}{\sum_{j\in S_t} n_j}\,\Delta_k^t. \]
Guarantee & limit
The aggregator sees only updates, never raw data ⇒ data-residency satisfied. But updates can leak via gradient inversion — combine with local DP-SGD for both guarantees.
finance use cases (1) Multi-bank joint fraud / AML model without sharing data; (2) cross-border improvement of a single global model.

Architectural patterns for compliant systems

Strategy 4: privacy is a property of the whole system, not just the model.
Strategy 4 · retrieval

Privacy-aware RAG: plug the three leaks

Retrieval-augmented generation leaks at three points: ingestion (embed & store), retrieval (the query embedding reaches the vector-DB operator), and inference (retrieved docs sit beside user data).

The systematic fix
  • Ingestion: anonymise before embedding; store only the anonymised version; keep the mapping in an access-controlled store.
  • Retrieval: compute embeddings with a local model — the query never leaves the network.
  • Inference: scan retrieved chunks for residual PII before prompting.
  • Access control: row-level security — each chunk carries an ACL; retrieval is filtered by the user's clearance.
why access control matters A junior analyst must not retrieve merger-target docs via an indirect question to the assistant.
Strategy 4 · governance

Audit trails and encrypted pipelines

Regulators expect full reconstruction of any client-affecting decision: who asked, what was retrieved, what was asked and answered, and what action followed.

Privacy-aware audit logging
Logging the full prompt + completion is the most complete record — but creates a secondary PII store under GDPR retention + DORA resilience. Better: log a hash of the prompt (integrity, not content) plus structured metadata — user ID, clearance, document IDs, model version, response time, outcome — to an immutable store.
Encryption end-to-end
At rest, in transit, ideally during inference.
  • Homomorphic encryption: compute on ciphertext — infeasible overhead for transformer inference today.
  • Confidential computing (TEE: Intel TDX, AMD SEV): practical; blocks even the cloud operator during processing.

Evaluation and compliance

Don't take privacy on faith — measure it, document it, and re-check it on a cadence.
Trust, but verify

Three quantitative tests to run after every fine-tune

the bigger picture Verify privacy — do not take it on faith from the training algorithm. Three standard quantitative tests, run after every fine-tune.
  1. Membership inference AUC — hold out fine-tuning docs as negatives; run the likelihood-ratio attack; measure the AUC of distinguishing members from non-members. 0.5 = perfect privacy; 1.0 = total failure. < 0.6 is commonly acceptable.
  2. Verbatim extraction rate — insert "canary" sensitive strings; measure the fraction recovered. DP fine-tuning ⇒ near-zero.
  3. Property inference accuracy — train a meta-classifier on shadow models to infer an aggregate property; compare to a random baseline.

Appendix — derivations, internals, and edge cases

Advanced composition, AWQ vs. GPTQ internals, re-identification as inference, and a RAG walkthrough. Beyond the core lecture.
Appendix A · the full bound

Advanced composition in full

The body slide quoted a simplified composition bound. Here is the full statement and why the simplification is safe.

For \(k\) mechanisms each \((\varepsilon,\delta)\)-DP, the composite is \((\varepsilon', k\delta+\delta')\)-DP for any \(\delta'>0\) with \[ \varepsilon' = \varepsilon\sqrt{2k\ln(1/\delta')} + k\varepsilon(e^{\varepsilon}-1). \] For small \(\varepsilon\) (\(e^{\varepsilon}-1\approx\varepsilon\)) the second term is \(O(k\varepsilon^2)\), often negligible vs. the first — giving the cited \(\varepsilon'\approx\varepsilon\sqrt{2k\ln(1/\delta')}\).
Basic composition uses post-processing immunity: any function of a DP output keeps the guarantee. Chain the two per-mechanism bounds via a union argument \(\Rightarrow (\varepsilon_1{+}\varepsilon_2,\,\delta_1{+}\delta_2)\). The advanced bound follows from the moments accountant tracking the Rényi divergence \(D_\alpha(P\|Q)=\frac{1}{\alpha-1}\log\sum_x P(x)^\alpha Q(x)^{1-\alpha}\) between output distributions on \(D, D'\) rather than the multiplicative distance.
Appendix B · quantisation internals

AWQ vs. GPTQ: why error is not uniform

Both compress 16-bit weights to 4-bit, but they disagree on where the danger lies. The 16→4-bit error is not spread evenly across weights.

AWQ's insight: ~1% of weights are salient (large activation magnitudes); aggressive quantisation of these causes large output errors. AWQ finds them from activation statistics on a small calibration set, then applies a per-channel scale transformation that protects them without raising bit width; the other 99% go uniform 4-bit with negligible loss.

GPTQ instead solves a block-wise reconstruction per weight matrix, minimising quantisation error given the Hessian of outputs with respect to weights.

Similar final perplexity; AWQ is faster (minutes vs. hours, 7B) and needs no GPU.
Appendix C · the formal threat model

Re-identification framed as an inference problem

Model re-identification as inference: the adversary observes the anonymised document and uses prior knowledge to form a posterior over who it is about.

The adversary observes the anonymised document \(\hat{d}\) and uses prior knowledge \(\mathcal{K}\) to form \[ \mathcal{P}(\text{individual} \mid \hat{d}, \mathcal{K}). \] Posterior concentrated on one individual \(\Rightarrow\) re-identification trivial. Posterior uniform over a large set \(\Rightarrow\) re-identification infeasible.
Assessment by simulation
Apply the pipeline, then attempt re-identification using the institution's own databases as \(\mathcal{K}\). Success ⇒ pipeline insufficient; strengthen generalisation / suppression. Failure on all test cases ⇒ adequate for that threat model. Point-in-time only — \(\mathcal{K}\) grows; re-review on a cycle.
Appendix D · putting it together

A privacy-aware RAG walkthrough, end to end

Credit analyst query: "Typical collateral structure for technology-sector LBO loans?"

  1. Query embedded locally by a self-hosted model.
  2. Top-5 chunks retrieved, filtered to the analyst's authorised set (policy docs, sector reports, anonymised historical loan files).
  3. Chunks scanned for PII — none (pre-anonymised at ingestion).
  4. Local LLM generates, citing retrieved content.
  5. Logged: timestamp, user ID, query hash (not the query), document IDs, model version → immutable audit store.
result No client-identifying information enters the prompt; no data crosses an external boundary.
