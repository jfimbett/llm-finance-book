01

Credit data: sources, privacy, and regulation

The richest behavioural dataset in finance — and one of the most heavily policed.
The founding cautionary tale

Why credit AI is different from everything else

Get a sentiment model wrong and you lose a trade. Get a credit model wrong and you face regulators, lawsuits, and headlines.

the Apple Card incident · 2019 David Heinemeier Hansson's Apple Card limit came in 20× lower than his wife's — despite joint tax returns and comparable profiles. Senator Warren wrote to Goldman Sachs; the NY DFS opened a formal investigation.
Credit data is two things at once
  • The richest behavioural dataset a bank has
  • One of the most legally regulated datasets in existence
Both directions carry consequences
  • Deny unfairly → Fair Lending violations
  • Extend recklessly → capital losses, systemic risk
  • Either → regulatory action, litigation, reputational damage
The raw material

What's actually inside a credit file

The three US bureaus — Equifax, Experian, TransUnion — each hold files on roughly 220 million adults. A standard file is four kinds of record.

  • Account data — 24–84 months of payment history per account, plus balance and limit
  • Inquiry data — hard inquiries (applications) vs. soft inquiries (pre-qualification)
  • Public records — bankruptcies (7–10 years), civil judgments
  • Collections — charged-off accounts

the structural limitation Bureau data is circular: it records credit behaviour, so you need credit to build credit. Roughly 45M US adults have thin or no file — recent immigrants, young adults, lower-income households.
Filling the gap — carefully

Alternative data can reach thin-file borrowers, within limits

When the bureau file is empty, lenders look elsewhere — but each new source comes with its own legal leash.

  • Bank transaction data — cash-flow signals; consent governed by Dodd-Frank §1033
  • Rent and utility payments — the largest recurring obligation, yet historically invisible to scores
  • Unstructured text — loan-purpose statements; "medical emergency" ≠ "consolidating payday loans"
  • Social / behavioural data — largely prohibited; too easily a proxy for protected classes
where the LLM earns its place That unstructured-text row is exactly where a language model adds signal a tabular scorecard can't.
The legal fence

Cleaning before modelling · 1 of 2

Two of the three universal preprocessing traps

Anonymisation
Quasi-identifiers create re-identification risk. Sweeney (2002): 87% of the US population is uniquely identifiable from ZIP + date of birth + sex. Fixes: tokenisation, generalisation (age → bracket), suppression, differential privacy.
Class imbalance
Default rates run 1–10% over 12 months. With 3% defaults in 1M rows, "always predict non-default" scores 97% accuracy and is useless. Re-weight the classes so each carries equal pull.
Give the minority class proportionally more weight so each class has equal aggregate influence on the gradient: \[ w_+ = \frac{n}{2\,n_+}, \qquad w_- = \frac{n}{2\,n_-}. \] Here \(n\) is the sample size and \(n_+, n_-\) the positive / negative counts. The class ratio no longer decides whose errors the optimiser cares about.
Cleaning before modelling · 2 of 2

Missing data isn't always missing by accident

The fact that a field is blank can itself be informative — a borrower with no bureau score is telling you something.

  • MCAR — missing completely at random; simple mean imputation is unbiased
  • MAR — missing at random; multiple imputation is consistent
  • MNAR — missing not at random; needs an explicit model of the missingness
The LLM move
Don't silently impute — serialise the gap into the text so the model can read it: "[INCOME: MISSING] [BUREAU_SCORE: 687]".

Turning a borrower into a prediction

Three ways to fine-tune for default prediction

First write the borrower's profile as a sentence; then pick how the model emits a verdict.

  • Sequence classification — a BERT encoder with a [CLS] head and sigmoid. The most common choice in published LLM credit-scoring papers.
  • Causal LM with label generation — prompt a decoder-only model, read off the label-token probability. Scales to few-shot.
  • Regression on probability — output a scalar probability directly.

Training without retraining everything

LoRA: fine-tune a giant model by nudging a tiny slice

Full fine-tuning updates every weight — expensive and storage-heavy. LoRA freezes the model and learns only a small low-rank correction, cutting trainable parameters dramatically.

Constrain the weight update to a low-rank factorisation: \[ \Delta W = B A, \quad B \in \mathbb{R}^{d \times r}, \; A \in \mathbb{R}^{r \times d}, \quad r \ll d. \] With \(r = 8\), \(d = 768\): about 48× fewer trainable parameters than full fine-tuning. The original \(W_0\) stays frozen; only \(A\) and \(B\) are trained.
Getting a number out of a word machine

How do you extract a single default probability from an LLM?

The problem
A decoder-only LLM produces a distribution over its entire vocabulary at each position — not a single scalar probability of default. So how do we read off a well-defined PD?

The trick: force the model to write its answer in a fixed format, then read the digits.

Using the outlines library:
  1. Define an output grammar — a JSON object with one key "pd" mapped to a float in \([0,1]\).
  2. At each step, intersect the model's next-token distribution with the grammar's automaton.
  3. Renormalise over valid tokens only:
\[ P_{\text{constrained}}(x_{t+1} = v \mid x_{1:t}) = \frac{\exp(\ell_v)\,\mathbb{1}[v \in \mathcal{V}_{\text{valid}}]} {\sum_{u \in \mathcal{V}_{\text{valid}}} \exp(\ell_u)}. \] The model writes {"pd": then produces the float digit by digit. If it emits 0.23, the implied default probability is \(\hat{p} = 0.23\).
Read that number carefully

A generated probability is not a calibrated one

the key point \(\hat{p}\) is not the model's "confidence" — it is the specific number the model was guided to express. It reflects the model's internal representation of risk, but it is not guaranteed to be calibrated against real default rates.
Making the number mean what it says

Calibration: of everyone scored 10%, do 10% actually default?

A probability is only useful if it matches reality. We check this with a reliability diagram, then fix it if the line bends.

Reliability diagram
Bin the interval \([0,1]\); plot mean predicted probability vs. observed default rate per bin. Perfect calibration is the 45-degree line.
Platt scaling
Fit a logistic regression on raw scores on a held-out set. Cheap; works with small data.
A model is perfectly calibrated if \[ \mathbb{E}[Y \mid \hat{P} = p] = p \quad \text{for all } p \in [0,1]. \] Platt scaling fits \[ \hat{p}_{\text{Platt}}(x) = \sigma\big(a \cdot f(x) + b\big), \] where \(a < 1\) corrects overconfident scores and \(a > 1\) corrects underconfident ones.
When the bend isn't a straight line

Isotonic regression: more flexible, hungrier for data

Platt assumes the miscalibration is a smooth logistic curve. Isotonic regression makes no such assumption — it just fits any non-decreasing staircase that best matches observed defaults.

Find the non-decreasing step function minimising squared error: \[ \hat{q} = \underset{q_1 \leq \cdots \leq q_n}{\arg\min} \sum_{i=1}^{n} (q_i - y_i)^2. \] Solved by the pool-adjacent-violators (PAV) algorithm in \(O(n)\) time. More flexible than Platt, but it needs more data.
03

Household decisions under uncertainty

Real people don't solve the optimisation — so what should an LLM do for them?
People are not the textbook agent

Why a "rational" borrower is the wrong default assumption

CFPB (2018)
One-third of US mortgage borrowers did not shop before accepting the first offer. The average non-shopper paid about \(\$\)300/year more in interest — over 30 years, tens of thousands of dollars.

Four reasons households depart from the neoclassical benchmark:

  1. Computational complexity — the full household problem is a high-dimensional stochastic control problem; people approximate
  2. Information asymmetry — knowledge of credit terms, taxes, insurance is required and mostly absent
  3. Dual-process cognition — high-stakes choices often made in fast, affective "System 1" mode
  4. Preference inconsistency — hyperbolic discounting, loss aversion, framing effects
A better model of the human

Households satisfice — and that changes the LLM's job

Simon's bounded rationality Households don't solve the optimisation; they satisfice — searching alternatives until one clears a threshold of acceptability, then stopping. The threshold itself adapts to recent outcomes.

So an LLM that just hands back the "optimal" choice misses the point. The value is in widening the set of alternatives a boundedly-rational household actually considers.

The right division of labour

What a decision-support agent must do — and not do

Three functions, ending with a deliberate restraint.

  1. Information extraction — parse term sheets and insurance exclusions into structured, comparable summaries
  2. Consequence tracing — compute the multi-period financial implications of each choice for this household
  3. Trade-off articulation — present options so the trade-offs are legible without nudging toward one
From principle to plumbing

A practical decision-support architecture

ComponentRole
Document ingestion layerRAG pipeline on term sheets
Household profile moduleIncome, wealth, risk preferences

design principle An LLM that always recommends the neoclassically optimal choice can undermine household autonomy when stated and revealed preferences diverge. The goal is extension, not replacement, of the household's cognitive frontier.

Persona agents and behavioural simulation

Conditioning an LLM on a backstory to preview how real borrowers would choose.
Silicon sampling

Give an LLM a backstory, and it answers like that demographic

Argyle et al. (2023)
An LLM conditioned on a detailed demographic backstory reproduces the survey-response distributions of the matching demographic group with striking fidelity. This is an empirical observation about distributional encoding — not a claim about sentience.

A persona is specified along four axes.

\(\Pi = (D, S, \Psi, L)\):
  • \(D\) — demographics: age, sex, household composition, education, region
  • \(S\) — socioeconomics: employment, income, wealth, existing debt
  • \(\Psi\) — psychology: risk aversion \(\gamma\), time preference \(\delta\), financial anxiety
  • \(L\) — financial literacy: the Lusardi–Mitchell 3-question battery

The same loan, reasoned two defensible ways

Maria → Option A

"Option B starts lower but says adjustable. If it goes to 18%… My rent is \(\$\)1,200 and some months I barely clear \(\$\)1,800. I can't risk my payment jumping when work is slow. At least I know exactly what I owe every month."

Thomas → Option B

"At 8%, five-year interest ≈ €4,300; at 12% ≈ €6,600. If rates stay below ~10% on average, B is cheaper. My wife's income covers fixed obligations. I'd choose B and set aside the monthly difference into the Livret A."

the observation Both responses are coherent and defensible. Neither is "wrong." The contrast reflects genuine heterogeneity in risk aversion, income volatility, and financial literacy.

Evaluation and model risk management

How good is the model — and can you defend it to a validator and a regulator?
Measuring discrimination power

The metrics credit people actually report

Credit modellers don't lead with accuracy — they ask how well scores separate defaulters from non-defaulters. Three measures do that, and they're tightly linked.

AUROC
good > 0.70 · excellent > 0.80
KS
good > 0.40 · excellent > 0.60
Gini
good > 0.40 · excellent > 0.60
AUROC is the probability a random defaulter outscores a random non-defaulter: \[ \text{AUROC} = P(S_+ > S_-). \] KS is the maximum gap between the two score CDFs: \[ \text{KS} = \sup_{s} |F_+(s) - F_-(s)|. \] Gini is an exact rescaling of AUROC (from the CAP/ROC relationship, not an approximation): \[ \text{Gini} = 2\cdot\text{AUROC} - 1, \] ranging 0 (random) to 1 (perfect). Retail credit models typically land at Gini 0.40–0.70.
The supervisory backbone

SR 11-7: an LLM is unambiguously a "model"

SR 11-7 (Fed / OCC, 2011) says every model must be conceptually sound, monitored, independently validated, and documented. LLMs qualify — no exemption. Three pillars, adapted.

Development
  • Document base-model provenance: data, licence, cutoff
  • Fine-tune data quality & vintage vs. current cycle
  • Watch for emergent, untested reasoning
Independent validation
  • Outcome analysis vs. a logistic baseline
  • Sensitivity to word order, synonyms, missing fields
  • Disparate-impact testing by protected class
  • Adversarial probing
Governance
  • Versioning: base-model updates may trigger revalidation
  • Prompt governance: a system-prompt change is a model change
  • Vendor risk: API models leave residual model risk
Explaining a single decision

SHAP turns one prediction into reason codes the law requires

FCRA demands specific reasons for an adverse action. SHAP assigns each feature a fair share of the prediction — borrowed straight from cooperative game theory — and the top contributors become the reason codes.

The SHAP value for feature \(j\) averages its marginal contribution over all feature subsets: \[ \phi_j(x) = \sum_{S \subseteq \mathcal{F} \setminus \{j\}} \frac{|S|!\,(|\mathcal{F}|-|S|-1)!}{|\mathcal{F}|!}\big[ f(S \cup \{j\}) - f(S) \big]. \] Three axioms pin it down: efficiency (\(\sum_j \phi_j = f(x) - \mathbb{E}[f(x)]\)), symmetry, and dummy.

Catching the model going stale

Drift monitoring: is today's population still last year's?

Models decay when the world moves. The Population Stability Index compares the distribution of inputs now against the baseline, and flags when they've drifted too far.

\[ \text{PSI} = \sum_{b=1}^{B} (p_b - q_b)\,\ln\!\left(\frac{p_b}{q_b}\right), \] summing over bins \(b\), where \(p_b, q_b\) are the current and baseline proportions. PSI \(< 0.1\): stable; \(0.1\text{–}0.25\): investigate; \(> 0.25\): model review required.
Keeping a human in the loop

Three review tiers — and the bias they guard against

GDPR Art. 22, ECOA, and SR 11-7 all require human oversight. Route decisions by difficulty.

Tier 1 · clear approvals
Random quality-control sample only.
Tier 2 · borderline
Active human review before the decision is communicated.
Tier 3 · appeals
Full decision record (incl. SHAP) reviewed; reviewer documents an independent assessment.
automation bias Reviewers anchor hard to a prominently displayed model recommendation. Mitigate: show the recommendation after independent assessment; present SHAP prominently; require documented reasoning before the reviewer sees the model's output.
Paper the regulator needs

The documentation package that ships with the model

  • MDR — Model Development Report: business need, conceptual framework, data, performance, limitations
  • MVR — Model Validation Report: independent validation findings and model-risk rating
  • GDPR Art. 22 — plain-language explanation for consumers, translated from SHAP via a constrained LLM
  • Model Inventory — versioned tracking of base model, fine-tune, calibration, and system prompt
Explaining a denial to a human

SHAP is the input to the explanation, never the output

GDPR Recital 71
The consumer has "the right to obtain human intervention, to express his or her point of view, and to contest the decision." The explanation must be comprehensible to someone with no financial or technical background.
Generated plain-language notice

"Your application for a €12,000 personal loan was assessed using an automated system… Our system estimated a higher probability of repayment difficulty than our lending criteria allow. Factors that increased your estimated risk: your credit bureau score of 672…; one late payment in the past twenty-four months; and a loan purpose of debt consolidation. Factors that partially reduced your estimated risk: six years of continuous full-time employment and an annual income of \(\$\)58,000. You have the right to request human review of this decision…"

key principle SHAP values feed the plain-language generator; they are not shown raw. Putting raw SHAP numbers in consumer communications creates confusion, not transparency.
Know your failure modes

Five ways an LLM credit model can fail you

  1. Prompt sensitivity — instability under word-order or synonym changes is a red flag; sensitivity analysis is a mandatory SR 11-7 step
  2. Calibration gap — constrained-decoded probabilities reflect internal representation, not empirical default rates; calibration is non-optional
  3. Protected-class proxies — historical text can encode correlations between language, geography, occupation and protected classes; test for intersectional disparate impact
  4. Adversarial manipulation — a sophisticated borrower may craft text to trigger favourable output; adversarial probing is part of validation
  5. Vendor lock-in & versioning — a silent base-model update changes outputs with no action by the institution; versioning contracts and revalidation triggers are required controls

Appendix — extended material

Utility elicitation, population simulation, and through-the-cycle validation. Beyond the core lecture.
Appendix A · utility theory

Framing the household choice as expected utility

Underneath "satisficing" sits the classical benchmark: a household picks the action whose expected utility of resulting wealth is highest. Risk aversion is one parameter, \(\gamma\).

The household with wealth \(W_0\) selects \[ a^* = \underset{k}{\arg\max} \; \mathbb{E}\big[U(W_0 + \Delta W_k)\big]. \] The most common parameterisation is CRRA utility: \[ U(W) = \frac{W^{1-\gamma}}{1-\gamma}, \quad \gamma > 0, \] where \(\gamma = 1\) is log utility, \(\gamma = 2\) moderate risk aversion, and \(\gamma > 5\) high risk aversion.
Appendix A · eliciting preferences

Letting the LLM measure a household's risk aversion

Rather than assume \(\gamma\), run a dialogue of lottery choices, find where the household is indifferent, and fit the utility curve to those points.

  1. Present a sequence of binary lottery choices to identify indifference points.
  2. Extract the stated indifference points from the conversation.
  3. Fit CRRA utility by minimising \[ \hat{\gamma} = \underset{\gamma}{\arg\min} \sum_{j=1}^{J} \big( U_\gamma(c_j) - \lambda_j U_\gamma(g_j) - (1-\lambda_j) U_\gamma(\varepsilon) \big)^2. \]
  4. Use \(\hat{\gamma}\) to evaluate the household's alternatives.
Quality hinges on the LLM running a coherent multi-turn dialogue without anchoring the responses.
Appendix B · population simulation

Running many personas to preview a product launch

Draw \(N\) personas from a target population and simulate their choices to see which subgroups respond to which product features — before any live test.

\[ \hat{P}(d = k) = \frac{1}{N} \sum_{i=1}^{N} \mathbb{1}[d_i = k], \] the share of \(N\) simulated personas choosing option \(k\).
  • Loan uptake — estimate take-up vs. APR before a live A/B test
  • Mortgage suitability — flag which products create distress risk for which persona types
  • Retirement planning — test auto-enrolment schemes for low-literacy households
Appendix B · calibration

A simulation is only useful if it matches a real survey

Validate the synthetic population against survey data (ECB HFCS, US SCF) before trusting it.

  1. Population alignment — post-stratification weights so the joint attribute distribution matches the target survey
  2. Behavioural alignment — compare simulated vs. survey decision distributions via a calibration loss
  3. Prompt refinement — iterate when calibration loss is high
\[ \mathcal{L} = \sum_k \big(\hat{P}_{\text{sim}} - \hat{P}_{\text{survey}}\big)^2. \]
the standard A validated simulation explaining 60–70% of cross-sectional variation is useful. An uncalibrated simulation that claims to predict population behaviour is not.
Appendix C · through-the-cycle validation

Models trained on good times underestimate the bad ones

Concept drift
The feature → default relationship shifts as the economy evolves. A model trained on 2015–2019 data may underestimate recession-period default risk.
Population drift
The applicant distribution shifts — new marketing strategy, regulatory change.

Vintage analysis: group loans by origination quarter; track AUROC and KS as outcomes mature; look for time trends in discrimination power.

Basel III Pillar 1 requires through-the-cycle (TTC) PD estimates — long-run average default rates — but LLMs trained on recent data tend to produce point-in-time (PIT) estimates. Convert by Platt scaling on a dataset spanning \(\geq 1\) full credit cycle (7–10 years): \[ \hat{p}_{\text{TTC}} = \sigma\big(a \cdot f(x) + b\big), \] where \(a\) and \(b\) absorb both the overconfidence correction and the cycle-position adjustment at once.
