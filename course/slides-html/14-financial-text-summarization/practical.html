Recap — the formulas you will need

Extractive vs. abstractive, ROUGE, and the three-step consistency recipe.
Two ways to summarize

Copy sentences, or write new ones?

Extractive
  • Picks verbatim sentences from the source
  • Faithfulness is free — it can only quote
  • Can read choppy
Abstractive
  • Generates new sentences
  • Fluent and readable
  • Hallucination-prone — can invent facts
the trade-off in one line Extractive trades fluency for safety; abstractive trades safety for fluency. In finance, the stakes decide.
Scoring a summary

ROUGE: how much of the reference did we recover?

ROUGE asks a recall question in plain English: of the words (or phrases) in the human reference summary, what fraction shows up in the machine's summary?

ROUGE-\(N\) (recall-oriented) counts matched \(n\)-grams: \[ \text{ROUGE-}N=\frac{\#\{\text{matched } n\text{-grams}\}}{\#\{n\text{-grams in reference}\}}. \] \(N=1\) is unigrams (single words), \(N=2\) is bigrams (adjacent word pairs).

ROUGE-L is LCS-based: it uses the longest common subsequence between candidate and reference, so it rewards words that appear in the same order without requiring them adjacent.
the bigger picture ROUGE is a word-overlap ruler. It is cheap and standard — but it does not understand meaning. Keep that in your pocket; Problem 1 exploits it.
Catching wrong numbers

The three-step consistency recipe

A summary can be fluent, well-scored, and still state a number that contradicts the filing. Consistency checking is the cheap gate that catches it.

  1. Extract each metric from multiple sources (prose narrative and the XBRL table).
  2. Normalise to a common unit and sign (e.g. all revenue in USD millions).
  3. Cross-reference — flag any pair that disagrees beyond a tolerance (e.g. 0.1%).
Faithfulness label set
verified matches the source  ·  unverifiable not in the source  ·  incorrect contradicts the source

Case study · the code · part 1a

Setup & helper functions

A short MD&A excerpt, plus three helpers: split into sentences, count words, measure cosine overlap.

import re, networkx as nx
from collections import Counter

MDNA = (
 "Revenue increased 13% to $56.5 billion in fiscal 2023. "
 "The growth was driven by strong cloud demand. "
 "Azure revenue grew 29% year over year. "
 "Operating margin expanded to 42%. "
 "Management expects continued momentum into fiscal 2024. "
 "The company faces foreign-exchange headwinds."
)

def sentences(text):
    return [s.strip() for s in re.split(r'(?<=[.!?])\s+', text) if s.strip()]

def bow(s):                       # bag-of-words vector
    return Counter(re.findall(r"[a-z]+", s.lower()))

def sim(a, b):                    # cosine over word counts
    common = set(a) & set(b)
    num = sum(a[w] * b[w] for w in common)
    den = (sum(v*v for v in a.values())**0.5) * \
          (sum(v*v for v in b.values())**0.5)
    return num / den if den else 0.0
Case study · the code · part 1b

TextRank: sentences vote for each other

def textrank(text, k=2):
    sents = sentences(text); vecs = [bow(s) for s in sents]
    G = nx.Graph()
    for i in range(len(sents)):
        for j in range(i+1, len(sents)):
            w = sim(vecs[i], vecs[j])
            if w > 0: G.add_edge(i, j, weight=w)
    scores = nx.pagerank(G)
    top = sorted(scores, key=scores.get, reverse=True)[:k]
    return [sents[i] for i in sorted(top)]

print(textrank(MDNA, k=2))
the idea Sentences vote for each other via cosine similarity; PageRank surfaces the most central (salient) ones — selected verbatim, so the extractive summary is faithful by construction.
Build a graph where node \(i\) is a sentence and edge weight \(w_{ij}=\mathrm{sim}(s_i,s_j)\) is the cosine of their bag-of-words vectors. PageRank assigns each sentence a stationary score \[ \pi_i = \frac{1-\delta}{n} + \delta \sum_{j} \frac{w_{ji}}{\sum_k w_{jk}}\,\pi_j, \] so a sentence is "important" if it is similar to many other important sentences. The top-\(k\) by \(\pi\) are the most central — the ones the document keeps coming back to.
Case study · the code · part 2

Consistency check: prose number vs. the table

# Table (XBRL-derived) ground truth, in USD millions
TABLE = {"revenue": 56500.0, "azure_growth_pct": 29.0}

# Numbers stated in the prose narrative
PROSE = {"revenue": 56500.0, "azure_growth_pct": 28.0}  # note: 28 vs 29

def consistency(table, prose, tol=0.001):
    flags = []
    for k in table.keys() & prose.keys():
        t, p = table[k], prose[k]
        rel = abs(t - p) / abs(t) if t else 0.0
        status = "OK" if rel <= tol else "MISMATCH"
        flags.append((k, t, p, f"{rel:.2%}", status))
    return flags

for row in consistency(TABLE, PROSE):
    print(row)
# -> ('revenue', 56500.0, 56500.0, '0.00%', 'OK')
# -> ('azure_growth_pct', 29.0, 28.0, '3.45%', 'MISMATCH')

Solution · Problem 1

ROUGE by hand — the scores

Reference \(R\) (7 tokens): revenue rose 13 percent to 56.5 billion.

Candidate A — "revenue rose 13 percent"
  • ROUGE-1 = \(4/7 \approx \mathbf{0.571}\)
  • ROUGE-2 = \(3/6 = \mathbf{0.500}\)
  • ROUGE-L = \(4/7 \approx \mathbf{0.571}\)
Candidate B — "profit fell 13 percent to 56.5 billion"
  • ROUGE-1 = \(5/7 \approx \mathbf{0.714}\)
  • ROUGE-2 = \(4/6 \approx \mathbf{0.667}\)
  • ROUGE-L = \(5/7 \approx \mathbf{0.714}\)
A matches \(\{\)revenue, rose, 13, percent\(\}\) → 4 unigrams of 7. Bigrams "revenue rose", "rose 13", "13 percent" → 3 of 6. The 4 words appear in order → LCS \(=4\).

B matches \(\{\)13, percent, to, 56.5, billion\(\}\) → 5 unigrams of 7. Bigrams "13 percent", "percent to", "to 56.5", "56.5 billion" → 4 of 6. LCS \(=5\).
Solution · Problem 1 · the lesson

The wrong-meaning summary won

Recall: A scored \(\approx 0.571\); B scored \(\approx 0.714\) on every ROUGE variant.

the lesson B outscores A on every variant despite inverting the meaning ("revenue rose" → "profit fell"). ROUGE rewards lexical overlap, not semantics or factual direction.
the move that follows This is exactly why finance needs factual-consistency / NLI-based checks (e.g. FactCC), not ROUGE alone, to catch a summary that says the opposite of the truth.

Which claim is most dangerous?

the net-income claim It is confidently stated, numerically wrong, and feeds directly into valuation / EPS reasoning — a decision-grade error wearing the clothes of a precise fact.

Wrap-up

Five takeaways — and where the next practical goes

Key takeaways

  • ROUGE is lexical — high overlap can coexist with inverted meaning; pair it with semantic (BERTScore) and factual (NLI / XBRL-grounded) checks.
  • Extractive buys faithfulness for free — TextRank quotes salient sentences via sentence-graph PageRank.
  • Consistency checking is a cheap, high-value gate — normalise, then cross-reference prose numbers against the XBRL table.
  • Tolerance matters — too loose hides material errors; too tight flags rounding noise.
  • Choose by the stakes — exact numbers / auditability → extractive; fluent human digests → abstractive (with checks).

Appendix · stretch problem

Hierarchical summarization of a 300,000-word S-1

An S-1 runs 300,000 words — far beyond any single context window. You cannot summarise it in one shot, so you summarise in stages.

  1. Stage 1: each of \(\sim\)30 sections → a 200–400-word summary. What is the approximate total intermediate token budget?
  2. Stage 2: concatenate the section summaries; a second-pass model → the final summary.
  3. Compare: cost vs. a retrieval-augmented approach that summarises only the top-\(k\) passages for "principal risk factors". When does each win?
30 sections \(\times\) ~300-word summaries \(\approx\) 9{,}000 words \(\approx\) 12{,}000 tokens of intermediate text, plus the input cost of reading all 300{,}000 words once. Retrieval skips most of the input read: it embeds and fetches only the passages matching the query, trading recall for cost when the question is narrow.
Appendix · stretch problem

Arithmetic via code — the ReAct principle

The lecture's ReAct principle: don't make the LLM do mental arithmetic — route the calculation to a code interpreter.

Given extracted revenue_2023 = 56500 and revenue_2022 = 50000 (USD M), an LLM is asked for YoY growth.

  1. Why might the LLM answer "~11%" or "15%" when computing inline?
  2. Write the one-line Python the agent should emit instead: (56500 - 50000) / 50000 * 100.
  3. State the verified result and explain how separating "decide-what-to-compute" from "actually-compute" eliminates the numerical-error class.
\[ \frac{56500 - 50000}{50000}\times 100 = \frac{6500}{50000}\times 100 = \mathbf{13\%}. \] The LLM decides the formula; the interpreter executes it exactly. The model never approximates a division, so the inline-arithmetic error class disappears.
